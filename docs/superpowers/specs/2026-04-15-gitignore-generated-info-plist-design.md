# Gitignore Generated Info.plist Files

Stop tracking `QuipMac/Info.plist` and `QuipiOS/Info.plist` in git so they can't be edited by hand and silently clobbered by the next `xcodegen generate`.

## Background

Both `QuipMac/Info.plist` and `QuipiOS/Info.plist` are generated outputs of `xcodegen`, produced from the `info.properties` section of each project's `project.yml`. But they're currently tracked in git, which creates a trap: a developer can edit the tracked Info.plist directly, see the fix land cleanly in `git diff`, and ship the commit — only to have the edit silently reverted the next time anyone runs `xcodegen generate`, which regenerates Info.plist from `project.yml` and overwrites hand-edits.

This exact trap bit us in commit `ed68292`: an earlier fix in `f7bb347` removed `NSAllowsLocalNetworking: true` from `QuipiOS/Info.plist` directly but left the flag in `QuipiOS/project.yml`. Every subsequent `xcodegen` run (for signing changes, version markers, etc.) silently re-added the flag, repeatedly breaking Tailscale connectivity from the iPhone until the bug was rediscovered and properly fixed at the `project.yml` layer. Today, `QuipiOS/project.yml` carries an explicit comment (lines 59–65) documenting this trap at its source of truth.

Wishlist reference: item #14 in `docs/superpowers/wishlist.md`.

## Overview

Untrack both `Info.plist` files and add explicit paths to `.gitignore`. Existing working copies keep their files on disk (they're regenerated from `project.yml` on every `xcodegen` run anyway). Fresh clones need to run `xcodegen generate`, which the README already documents.

Deliberately narrow scope:

- **What is in this spec:** untrack two specific `Info.plist` files, add two explicit paths to `.gitignore`, add a short README note explaining the rule and its asymmetry with `.xcodeproj`.
- **What is NOT in this spec:**
  - Untracking `.xcodeproj/project.pbxproj` (also xcodegen-generated but tracked). The risk profiles differ: `Info.plist` is 60 lines of legible XML that looks editable, while `.xcodeproj` is 10K+ lines of generated UUID-soup nobody would hand-edit. Asymmetric rules are acceptable when the risks are asymmetric.
  - Adding a pre-commit hook or CI check that surfaces drift. Rejected — doesn't *prevent* the trap, only reports it after the fact, and there's no existing CI in the repo (no `.github/workflows/`) so adding one is a much larger scope than this item.
  - Rewriting git history to scrub old `Info.plist` versions. `git blame` and historical spelunking stay intact.
  - Writing a drift-detection helper script. The two-command manual check (`xcodegen generate && git diff`) is simple enough to not warrant automation.

## Git Changes

Single commit on `eb-branch`. Four file touches:

1. **`git rm --cached QuipMac/Info.plist QuipiOS/Info.plist`** — stops tracking, leaves files on disk.
2. **`.gitignore`** — add two explicit path entries (not a glob), with a comment pointing at the README:
   ```
   # Generated by xcodegen from <project>/project.yml — see README
   QuipMac/Info.plist
   QuipiOS/Info.plist
   ```
3. **`README.md`** — short note near the existing "Building" section (around lines 42–50 where `xcodegen generate` is already mentioned), explaining:
   - Both `Info.plist` files are gitignored.
   - They are regenerated from `project.yml` by `xcodegen generate`.
   - Edit `project.yml`, not the `Info.plist`.
   - `.xcodeproj` files are *not* gitignored — so fresh clones can open in Xcode without a mandatory `xcodegen` first — but `Info.plist` is small enough to fall into "I'll just edit it directly" traps, which is why the rule is asymmetric.

## Pre-Commit Safety Gate

Before the commit lands, the developer must verify there is no drift between the currently-tracked `Info.plist` files and what `xcodegen generate` produces right now. If there is drift, the currently-tracked file contains a hand-edit that would be lost the moment we untrack — that edit must be moved to `project.yml` in a *separate preceding commit*, then the untrack commit follows.

Verification procedure:

```bash
cd QuipMac  && xcodegen generate && cd ..
cd QuipiOS  && xcodegen generate && cd ..
git diff -- QuipMac/Info.plist QuipiOS/Info.plist
```

Expected result: empty diff. If the diff is non-empty, stop, inspect, and reconcile at the `project.yml` layer before proceeding.

(Manual visual inspection of both `Info.plist` files against their `project.yml` sources during brainstorming showed no drift — every key present in `project.yml` matches the tracked file — but the real check runs `xcodegen`, not an eyeball, because xcodegen may inject or reorder keys in ways that aren't obvious from reading.)

## Testing

- **Untracked status.** `git ls-files | grep Info.plist` returns nothing.
- **Regenerability.** In a clean tree, run `xcodegen generate` in both project directories. `git status` remains clean afterward (the regenerated files are gitignored).
- **Builds still work.** Open both Xcode projects and build each scheme. Confirm no errors referencing missing `Info.plist`.
- **History preserved.** `git log -- QuipMac/Info.plist` still shows all pre-untrack history. `git blame` on the last tracked revision of the file works. This is a sanity check that the commit stopped at untracking and did not accidentally rewrite history.

No unit tests — this is a git-configuration change with no runtime behavior to exercise.

## Risks and Known Unknowns

- **jboert's workflow on upstream merge.** If this commit reaches `main`, jboert's next `xcodegen` run will leave `Info.plist` files in his working tree in an untracked-but-present state. This is the intended new normal, but it may look unexpected to him the first time. The commit message should be explicit that this is a hygiene change. No technical breakage is expected — his existing workflow still works, the files still exist, nothing refers to them through git.
- **Fresh-clone Xcode workflow.** Anyone who clones the repo without running `xcodegen` first will open the project in Xcode and see missing `Info.plist` files. The README already instructs `xcodegen generate` as the first build step, so this only affects someone skipping the README. Low probability, low blast radius — worst case they see an error, re-read the README, run xcodegen, and move on.
- **Future re-tracking by mistake.** If someone later runs `git add -A` and the `.gitignore` entries are missed or removed, the files could silently re-enter tracking. Mitigation: the `.gitignore` entries carry a comment pointing at the README, and the README explains the rule. This is the standard hygiene level for `.gitignore`-based discipline — no further guardrail.
- **Unrelated `Info.plist` files added later.** If a future sub-project or third-party integration adds its own `Info.plist` somewhere (e.g., `QuipMacTests/Info.plist`), it will not be auto-ignored — the `.gitignore` entries are explicit paths, not a glob. That's intentional: explicit paths prevent accidental ignoring of non-xcodegen `Info.plist` files. A future developer who needs to ignore a third `Info.plist` just adds its path.
